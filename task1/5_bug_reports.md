# Баг-репорты

## БАГ-001: При успешной приёмке груза статус ЭТрН не обновляется на «Груз принят грузополучателем»

| **ID** | BUG-001 |
|--------|---------|
| **Заголовок** | После подписания грузополучателем `ON_TRNVPRGRPO` (СодОпер=1) статус накладной остаётся «В пути» |
| **Серьёзность** | Major |
| **Приоритет** | High |
| **Окружение** | Тестовый стенд оператора ЭДО, версия API v2.3 |
| **Предусловия** | Создана ЭТрН, подписаны Титул 1 и Титул 2, груз доставлен. |

**Шаги воспроизведения**
1. Грузополучатель открывает ЭТрН, переходит на вкладку «Приёмка груза».
2. Заполняет все обязательные поля (фактические даты, массу, количество мест, состояние груза).
3. Нажимает «Подписать и отправить».
4. После успешного уведомления о подписании проверяет статус ЭТрН в списке документов.

**Фактический результат**  
Статус документа остаётся «В пути». В логах ошибок нет, XML-файл сформирован корректно, подпись валидна.

**Ожидаемый результат**  
Статус должен измениться на «Груз принят грузополучателем» и должна начаться процедура регистрации в ГИС ЭПД.

**Дополнительная информация**  
При просмотре деталей накладной в поле `СодОпер` отображается значение «1». Ошибка воспроизводится для всех типов грузов. Временно обходной путь отсутствует.

## БАГ-002: XSD-схема `ON_TRNVPRPRV.xsd` не допускает множественные оговорки `ЗамСостГруз` при заполнении элемента `ЗамПрвПрием`

| **ID** | BUG-002 |
|--------|---------|
| **Заголовок** | Невозможно указать несколько кодов оговорок (`ЗамСостГруз`) в одном экземпляре `ЗамПрвПрием` |
| **Серьёзность** | Minor |
| **Приоритет** | Medium |
| **Окружение** | Система формирования ЭТрН, генератор XML на основе XSD 5.01. |

**Шаги воспроизведения**
1. Перевозчик при оформлении Титула 2 хочет указать две оговорки: «05 – упаковка груза повреждена» и «10 – нарушение пломбы».
2. В интерфейсе выбирает оба значения.
3. Система генерирует XML с двумя элементами `<ЗамСостГруз>05</ЗамСостГруз>` и `<ЗамСостГруз>10</ЗамСостГруз>` внутри одного `<ЗамПрвПрием>`.
4. Выполняется валидация по XSD.

**Фактический результат**  
XSD (фрагмент):
  xml
<xs:element name="ЗамСостГруз" minOccurs="0" maxOccurs="unbounded">

*Описание ситуации*
Теоретически допускается использование атрибута `maxOccurs="unbounded"`.  
Однако текущая схема XSD спроектирована таким образом, что после проверки через **Schematron** возникает ошибка.

*Причина ошибки*
Конфликт с правилом, которое требует наличия **единственного** элемента `ИнЗамСостГруз` при условии, что `ЗамСостГруз = 12`.

*Наблюдаемое поведение*
- При указании **двух кодов** (не равных `12`) валидация проходит успешно.
- Однако бизнес-логика системы ожидает **один код**, тогда как фактические данные требуют указания **множественных кодов**.

---

**Ожидаемый результат**
1. Возможность указывать **несколько оговорок (кодов)** в одном экземпляре элемента `ЗамПрвПрием`.
2. Корректная валидация как по XSD, так и по Schematron.
3. **Альтернатива**: явно задокументировать, что допускается только одна оговорка, а все комбинации должны передаваться текстовым полем `ИнЗамСостГруз`.

---

*Дополнительная информация (из формата)*
Согласно описанию формата (п. 5, таблица 5.5):
- Поле `ЗамСостГруз` **может повторяться** (признак множественности — «НМ»).

Однако:
- Реализация **XSD** и правила **Schematron** противоречат этому описанию.

---

*Предлагаемые решения*
1. **Исправить XSD** — убрать ограничение `maxOccurs="1"`, приведя схему в соответствие с описанием формата.
2. **Изменить бизнес-логику** — использовать один код, а все дополнительные сведения передавать в виде комбинированного текста в поле `ИнЗамСостГруз`.

---

*Рекомендация*
Предпочтительным является **первый вариант** (корректировка XSD), так как он сохраняет семантику множественных кодов и соответствует официальному описанию формата.
