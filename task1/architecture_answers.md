# Архитектурные кейсы тестирования ЭТрН

## 1. Тестирование статусной модели
- **Конечные автоматы**: статусы ЭТрН: «Черновик» → «Ожидает отгрузки» → «Груз принят перевозчиком»/«В пути» → «Груз принят грузополучателем» (или «Отказ», «Частично принят»).  
- **Проверки**: переходы только по разрешённым событиям (например, из «Ожидает отгрузки» нельзя сразу перейти в «Груз принят грузополучателем»).  
- **Согласованность**: после каждого перехода проверять, что все обязательные документы подписаны и зарегистрированы в ГИС ЭПД.  
- **Нагрузочное тестирование**: массовое изменение статусов (сотни/тысячи накладных) – проверка производительности БД и очередей.

## 2. Интеграция с ГИС ЭПД
- **API-взаимодействие**: имитация ГИС ЭПД через моки или тестовый контур. Проверить:
  - Отправка Титулов 1–3: корректность формата, обязательность полей.
  - Получение УИД от ГИС ЭПД.
  - Обработка ошибок от ГИС (например, «Неверный формат», «Дубликат»).
- **Асинхронность**: подтверждение регистрации может приходить с задержкой. Тестировать сценарии повторных запросов, таймаутов, retry-механизмов.
- **Безопасность**: аутентификация и авторизация при вызове API ГИС ЭПД (токены, сертификаты).

## 3. Электронные подписи (УКЭП, простая, неквалифицированная)
- **Проверка подписей**: для каждого файла обмена требуется подпись отправителя. Тестировать:
  - Подпись корректным сертификатом – успех.
  - Подпись чужим/просроченным/отозванным сертификатом – ошибка.
  - Отсутствие подписи – документ не принимается.
- **Доверенности**: при `СпосПодтПолном=3` (доверенность в машиночитаемом виде) проверять наличие и валидность доверенности, ссылку на ИС хранения.
- **Производительность**: время подписания больших XML-файлов (до 100 Мб).

## 4. Масштабируемость оператора ЭДО
- Нагрузочное тестирование: одновременная отправка большого количества ЭТрН (например, 1000 накладных в минуту). Проверка:
  - Отсутствие потери сообщений.
  - Время обработки в очереди.
  - Использование ресурсов (CPU, память, сеть).
- Тестирование отказоустойчивости: падение одного из узлов оператора ЭДО – обработка должна продолжиться на другом узле (replication, failover).

## 5. Асинхронная обработка и уведомления
- **Механизмы**: очереди (RabbitMQ/Kafka), веб-хуки.
- **Проверки**:
  - После отправки документа участник получает уведомление о новом статусе.
  - Уведомления доставляются при сетевых сбоях (повторные попытки).
  - Идемпотентность: повторная отправка одного и того же уведомления не приводит к дублированию событий.
- **Тайминги**: задать максимальное время ожидания уведомления (SLA).

## 6. Обработка ошибок и восстановление
- **Валидация XSD + Schematron**: на границе входа (шлюз оператора ЭДО) должна происходить строгая валидация. Отклонённые документы попадают в очередь ошибок с кодом причины.
- **Логирование**: каждый шаг (получение, валидация, подпись, отправка в ГИС ЭПД) должен логироваться с traceId для трассировки.
- **Мониторинг**: метрики (количество успешных/ошибочных ЭТрН, время обработки, задержки вызовов ГИС ЭПД).

## 7. Тестирование изменяемых данных (доп. файлы обмена)
- Процессы изменения грузополучателя/порта (`ON_TRNVPRPEREADR`), замены судна (`ON_TRNVPRZAMEN`), финансового состояния (`ON_TRNVPRFINSOST`).  
- **Проверки**:
  - Ссылка на исходный УИД/файл должна существовать.
  - Новая версия документа не должна перезаписывать старую, а создавать новую запись с историей изменений.
  - Возможность отката или исправления (`ИспрТип` с `ИдПредФайлИспр`).

## 8. Безопасность и конфиденциальность
- Разграничение доступа: грузоотправитель не должен видеть финансовые данные перевозчика до их подписания и т.д.
- Шифрование каналов связи (TLS).
- Проверка, что чувствительные данные (ИНН, КПП, расчётные счета) не попадают в открытые логи.

## 9. Тестирование XSD-схем и их версионирование
- Версия формата 5.01. При выпуске новой версии убедиться в обратной совместимости.
- Автоматизированная проверка всех примеров XML из документации на соответствие XSD (тесты конформности).
