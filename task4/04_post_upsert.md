# 4. POST `/api/v2/orders/{id:uuid}` – создание заказа с клиентским ID (upsert)

## Описание
Создаёт заказ с указанным клиентом идентификатором. Если заказ с таким ID уже существует, выполняет его обновление (upsert).  
Поведение соответствует идемпотентной операции.

## Тестовые сценарии

| № | Сценарий | Ожидаемый HTTP‑статус | Пояснение |
|---|----------|------------------------|-----------|
| 4.1 | Успешное создание заказа с указанным ID | **201 Created** | Ресурс создан с предоставленным клиентом ID |
| 4.2 | Заказ с указанным ID уже существует — обновление | **200 OK** или **204 No Content** | Заказ обновлён; `200` — если возвращается read‑model, иначе `204` |
| 4.3 | Конфликт: ID уже существует, но обновление запрещено политикой API | **409 Conflict** | Конфликт идентификаторов |
| 4.4 | Некорректный формат UUID в пути | **400 Bad Request** | Ошибка валидации параметра пути |
| 4.5 | Тело запроса не соответствует схеме | **400 Bad Request** | Ошибка валидации тела запроса |
| 4.6 | Отсутствует или недействителен токен авторизации | **401 Unauthorized** | Требуется аутентификация |
| 4.7 | Недостаточно прав для создания/обновления заказа | **403 Forbidden** | Доступ запрещён |
| 4.8 | Внутренняя ошибка сервера / сбой БД | **500 Internal Server Error** | Неожиданная ошибка |

## Примечания
- При обновлении (п. 4.2) предпочтительно использовать `204 No Content`, если не требуется возвращать обновлённые данные.
- `409 Conflict` используется только в случаях, когда API принципиально запрещает обновление через этот метод.
